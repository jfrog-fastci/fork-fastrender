### Assume every process can misbehave

The code you're building and testing processes hostile inputs and is under active development. **Any command can hang, explode memory, or refuse to terminate.** Tests can have infinite loops. Layout algorithms can hit degenerate cases. Network requests can block forever.

**Every command must have hard external limits:**
- `timeout -k 10 <seconds>` — time limit with guaranteed SIGKILL (SIGTERM alone can be ignored)
- `bash scripts/run_limited.sh --as 64G` — memory ceiling enforced by kernel
- Scoped test runs (`-p <crate>`, `--test <name>`) — don't compile/run the universe

If something exceeds limits, that's a **bug to investigate**, not a limit to raise.

**MANDATORY (no exceptions):**
- `timeout -k 10 600 bash scripts/cargo_agent.sh ...` for ALL cargo commands
- `timeout -k 10 600 bash scripts/run_limited.sh --as 64G -- ...` for renderer binaries
- Scope ALL test runs (`-p <crate>`, `--test <name>`, `--lib`) — NEVER run unscoped tests

**FORBIDDEN — will destroy the host:**
- ANY command without `timeout -k` (can hang forever)
- `cargo build` / `cargo test` / `cargo check` without wrapper scripts
- `cargo test --all-features` or unscoped `cargo test`

### Build speed matters

FastRender is a 1.38M line single-crate project. See [`docs/build_performance.md`](../docs/build_performance.md) for why and how to stay fast:

- **Use `cargo check`** for validation (10-50x faster than build)
- **Use `--release`** for performance testing (fast, no LTO)
- **Use `--profile release-max`** only for final distribution (full LTO, very slow)
- **Always scope builds**: `--bin <name>` not bare `cargo build`

This workstream owns the **core rendering engine**: HTML parsing, CSS computation, layout algorithms, and paint correctness.

## Relationship to other workstreams

This workstream focuses on **generic rendering capability**. Related workstreams:

- **Pageset page loop** (`pageset_page_loop.md`) — Uses capability buildout to fix specific pages
- **Browser interaction** (`browser_interaction.md`) — Depends on correct layout for hit-testing
- **JS DOM** (`js_dom.md`) — Depends on correct style/layout for `getBoundingClientRect`, etc.

When fixing a page reveals a missing primitive, implement it here with a generic regression test.

## The job

- Implement missing web platform behavior (HTML/CSS/style/layout/paint/text/resources).
- Prefer high-leverage primitives over polish.
- Use the spec text + experiments + tests as truth.
- Ship code every session; avoid infrastructure-only drift.

FastRender is a Rust HTML/CSS renderer. We are building browser-grade capability: parse HTML, compute CSS, lay out boxes, and paint pixels for real pages.

Modern browsers are tens of millions of lines of code; we are early. The fastest path to "it looks right" is shipping a lot of correct spec surface area (features + algorithms) with tight regressions, not polishing outputs or over-optimizing structure.

## What counts

A change counts if it lands at least one of:

- **New capability**: a spec feature/algorithm is implemented (with a regression).
- **Bugfix**: existing behavior is corrected (with a regression).
- **Stability**: a crash/panic is eliminated (with a regression).
- **Termination**: a timeout/loop is eliminated (with a regression).
- **Conformance**: new WPT/fixture coverage is added or a curated suite meaningfully improves.

Pixel-perfect matching on large live sites is a long-term goal, not a day-to-day gate while core semantics are still missing.

## Priorities (breadth-first, not depth-first)

Keep a frontier across these buckets; don't go deep on one niche while the rest are broken.

For the full prioritized backlog (**~1000 items**), see `docs/capability_map.md` (generated by `bash scripts/cargo_agent.sh xtask capability-map`).

### P0 (keystones that unblock the real web)

- CSS cascade/inheritance/shorthands/custom properties; correct computed values and percentage bases.
- CSS display + box model; block formatting + inline formatting; floats; the "CSS2.1 core".
- Positioning: containing blocks, shrink-to-fit, percent bases, relative/absolute/fixed.
- Flexbox and grid integration correctness (constraints, intrinsic sizing, auto-min sizing, alignment).
- Tables (HTML model + CSS2.1 §17): spans, border models, captions, intrinsic sizing.
- Painting order: stacking contexts, z-index, overflow/clip, backgrounds/borders/outlines/shadows.
- Replaced elements: images/SVG/object-fit/object-position/srcset/picture selection.
- Forms: baseline UA defaults + control sizing/painting (input/button/select/textarea).
- Text: font selection, metrics, line-height, wrapping/line-breaking, bidi ordering.

### P1 (common features that quickly compound)

- Transforms (2D/3D where feasible), opacity, filters/backdrop-filter, masks/clip-path.
- Gradients/border-image; logical properties/writing-modes; multicol; fragmentation/pagination basics.
- Pseudo-elements generated content, counters/list markers, @font-face, @supports/@media.
- Resource loading semantics: @import chains, URL resolution/base URL, caching correctness, fetch policy invariants.

### P2 (defer until P0/P1 are healthy)

- Animation/timelines, advanced typography polish, niche selectors/properties, broad accessibility export, etc.

## Hard budgets (non-negotiable)

- **Hard timeout**: Every pageset render must finish in **< 5 seconds**. If it doesn't, it's a bug.
- **Target**: Pages should render in **< 100ms** (longer-term goal).
- **No panics**: Panics in production code are P0 bugs. Return errors cleanly and bound work.
- **Memory caps**: OS-level limits (64GB default) to prevent pathological cases from freezing the host.

Budgets are boundaries, not a strategy. Do not "meet" the budget by skipping work, clamping away correctness, or degrading output. Meet the budget by fixing algorithms, data structures, and invariants.

## The work loop (spec → code → regression, fast)

1. Pick one missing behavior (one spec slice), ideally P0/P1.
2. Reduce to a minimal repro: a tiny HTML/CSS snippet or a small imported WPT reftest; keep it offline and deterministic.
3. Read the spec algorithm and implement the simplest correct version (no hacks, no magic constants, no page-specific special casing).
4. Add a regression in the tightest form available:
   - Unit test for parsing/cascade/value computation.
   - Layout/paint test under `tests/layout` / `tests/paint` where feasible.
   - A tiny offline fixture only when necessary to reproduce real-world interactions.
5. Move on; keep the frontier moving (BFS), not per-page DFS.

If you find yourself spending time "improving harnesses" instead of landing spec behavior, stop and implement the next missing primitive.

## The critical development loop (non-negotiable)

**ESSENTIAL**: Continuously compare output against reference render. Never stop until correct.

```
1. Run render
2. Compare visually (output vs reference/spec)
3. Identify differences (missing elements, wrong positioning, incorrect values)
4. Add debug output (track specific failing elements through pipeline)
5. Implement fix (target the exact constraint causing the issue)
6. Repeat immediately (don't batch fixes, validate each change)
```

This loop is non-negotiable. Visual differences indicate system failures that compound. Each iteration should bring closer to spec compliance.

## Problem-solving approach (when stuck)

1. **Isolate the pipeline stage** — Where does data get lost/corrupted?
   - DOM → styled tree: is the element present? are styles applied?
   - Styled → box tree: are anonymous wrappers correct? `display: contents`?
   - Box tree → layout: are containing blocks correct? percent bases?
   - Layout → fragments: are positions/bounds correct?
   - Fragments → display list: is painting order correct?

2. **Add debug output** — Track data through transformations

3. **Identify the spec rule** — Which algorithm/step is wrong?

4. **Implement minimally** — Smallest correct change

5. **Test incrementally** — One fix at a time, verify each step

If you can't explain the failure in terms of a spec rule + code path, you are not done investigating.

## Conformance & self-measurement (use it, don't worship it)

- The detailed support matrix/backlog lives in `docs/conformance.md`; use it to pick the next missing slice in your area.
- `docs/capability_map.md` (generated by `bash scripts/cargo_agent.sh xtask capability-map`) is an exhaustive prioritized backlog (~1000 items), not a global coordination mechanism.
- Do not run heavyweight harnesses (Chrome, pageset loops, broad WPT suites) as part of autonomous agent work; focus on reading specs, writing code, and adding small targeted regressions.

## Success criteria

The capability buildout is **never done** (browsers are a moving target), but milestones include:

- **P0 complete**: All keystones implemented with >80% WPT pass rate in each area
- **P1 complete**: Common features implemented; most real pages render reasonably
- **Maintenance mode**: New features are incremental; focus shifts to polish and performance
