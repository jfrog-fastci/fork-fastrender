# Workstream: capability buildout (spec-first)

Common repo-wide rules (non-negotiables, resource limits, disk hygiene) live in `AGENTS.md`. This
doc is only the **capability buildout** workstream.

FastRender is a Rust HTML/CSS renderer. We are building browser-grade capability: parse HTML, compute CSS, lay out boxes, and paint pixels for real pages.

Modern browsers are tens of millions of lines of code; we are early. The fastest path to “it looks right” is shipping a lot of correct spec surface area (features + algorithms) with tight regressions, not polishing outputs or over-optimizing structure.

## The job

- Implement missing web platform behavior (HTML/CSS/style/layout/paint/text/resources).
- Prefer high-leverage primitives over polish.
- Use the spec text + experiments + tests as truth.
- Ship code every session; avoid infrastructure-only drift.

## What counts

A change counts if it lands at least one of:

- **New capability**: a spec feature/algorithm is implemented (with a regression).
- **Bugfix**: existing behavior is corrected (with a regression).
- **Stability**: a crash/panic is eliminated (with a regression).
- **Termination**: a timeout/loop is eliminated (with a regression).
- **Conformance**: new WPT/fixture coverage is added or a curated suite meaningfully improves.

Pixel-perfect matching on large live sites is a long-term goal, not a day-to-day gate while core semantics are still missing.

## Priorities (breadth-first, not depth-first)

Keep a frontier across these buckets; don’t go deep on one niche while the rest are broken.

For the full prioritized backlog (**~1000 items**), see `docs/capability_map.md` (generated by `cargo xtask capability-map`).

### P0 (keystones that unblock the real web)

- CSS cascade/inheritance/shorthands/custom properties; correct computed values and percentage bases.
- CSS display + box model; block formatting + inline formatting; floats; the “CSS2.1 core”.
- Positioning: containing blocks, shrink-to-fit, percent bases, relative/absolute/fixed.
- Flexbox and grid integration correctness (constraints, intrinsic sizing, auto-min sizing, alignment).
- Tables (HTML model + CSS2.1 §17): spans, border models, captions, intrinsic sizing.
- Painting order: stacking contexts, z-index, overflow/clip, backgrounds/borders/outlines/shadows.
- Replaced elements: images/SVG/object-fit/object-position/srcset/picture selection.
- Forms: baseline UA defaults + control sizing/painting (input/button/select/textarea).
- Text: font selection, metrics, line-height, wrapping/line-breaking, bidi ordering.

### P1 (common features that quickly compound)

- Transforms (2D/3D where feasible), opacity, filters/backdrop-filter, masks/clip-path.
- Gradients/border-image; logical properties/writing-modes; multicol; fragmentation/pagination basics.
- Pseudo-elements generated content, counters/list markers, @font-face, @supports/@media.
- Resource loading semantics: @import chains, URL resolution/base URL, caching correctness, fetch policy invariants.

### P2 (defer until P0/P1 are healthy)

- Animation/timelines, advanced typography polish, niche selectors/properties, broad accessibility export, etc.

## The work loop (spec → code → regression, fast)

1. Pick one missing behavior (one spec slice), ideally P0/P1.
2. Reduce to a minimal repro: a tiny HTML/CSS snippet or a small imported WPT reftest; keep it offline and deterministic.
3. Read the spec algorithm and implement the simplest correct version (no hacks, no magic constants, no page-specific special casing).
4. Add a regression in the tightest form available:
   - Unit test for parsing/cascade/value computation.
   - Layout/paint test under `tests/layout` / `tests/paint` where feasible.
   - A tiny offline fixture only when necessary to reproduce real-world interactions.
5. Move on; keep the frontier moving (BFS), not per-page DFS.

If you find yourself spending time “improving harnesses” instead of landing spec behavior, stop and implement the next missing primitive.

## Conformance & self-measurement (use it, don’t worship it)

- The detailed support matrix/backlog lives in `docs/conformance.md`; use it to pick the next missing slice in your area.
- `docs/capability_map.md` (generated by `cargo xtask capability-map`) is an exhaustive prioritized backlog (~1000 items), not a global coordination mechanism.
- Do not run heavyweight harnesses (Chrome, pageset loops, broad WPT suites) as part of autonomous agent work; focus on reading specs, writing code, and adding small targeted regressions.
